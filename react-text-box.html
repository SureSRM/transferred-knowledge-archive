React JS es un framework para la realización de aplicaciones frontend .

Pre-requisitos

  • Node v8.10.0

  • Node Package Manager v5.6.0

Introducción

Uno de los problemas a los que se enfrenta una aplicación web que obtiene (fetch) datos de un servidor es el retardo con el que éstos mismos llegan así como la posibilidad de inconsistencia de los resultados debido a que pueden no llegar ordenados.

Puesta en marcha

Para la puesta en marcha del śistema debemos ejecutar desde el directorio raíz los comandos de instalación y arranque del servidor web y del servicio backend.

Servidor Web

Es el encargado de servir las páginas al navegador web. En este caso, se usará Node ya que viene empaquetado en la platilla que ofrece create-react-app.

$ npm install && npm start
Servicio Backend

Para simular un servicio utilizaremos json-server. Un servidor ad-hoc escrito en Javascript que permite crear servicios REST rápidamente con una sola operación.

$ npx json-server \ (1)
--watch suggestions.json \ (2)
--port 3001 \ (3)
--delay 500 (4)
1 npx <package-name> <args> es un comando del Node Package Manager que ofrece descargar, ejecutar y eliminar la final un módulo de npm. Es muy útil en contraposición a npm install -g <package-name> && <package-name> <args> si lo que se desea es ejecutarlo una vez y no dejar software instalado en la máquina.
2 El fichero que utilizará json-sevrer como fuente de datos. Se puede editar para añadir más palabras, si uno lo desea.
3 El puesto en el expone el servicio
4 El delay con el que responderá a las peticiones. Es imprescindible este punto ya que queremos observar el retardo en el frontend.

El fichero que utilizaremos como fuente de datos es:

suggestions.json
{
  "words":[
    {"value": "abba"},
    {"value": "abuela"},
    {"value": "aceituna"},
    {"value": "acceso"}
  ]
}

Las soluciones

Con los dos servicios node y json-server podemos acceder a la dirección donde el servidor de node está escuchando para responder la página: localhost:3000

En ella veremos tres buscadores distribuídos en 3 columas:

  • Search Box Unsafe

  • Search Box Safe 1

  • Search Box Safe 2

Cada una de ellas se corresponde a una implementación distinta del mismo producto: una barra de búsqueda con función de sugerencias.

Searchbox Unsafe

Esta es la solución insegura debido a que no hace ningún tipo de comprobación en la respuesta. Esto puede llevar a situaciones en las que llegue una respuesta tan tarde, que el estado de la aplicación haya cambiado y ya no sea deseable esa respuesta.

El paso clave es de este proceso, no son aquellos estados donde la consistencia es LOADING. Es inevitable una inconsistencia de ese tipo, ya que la información no llega inmediata.

El paso clave es el WRONG, porque siendo el estado de la aplicación: {valor-de-la-caja-de-texto, sugerencias} consistente en el paso anterior {ab, sugerencias-de-ab} cambia sólo y sin interacción del usuario a un estado inconsistente {ab, sugerencias-de-abu}.

Esto se debe a que la respuesta ha llegado tarde, pero el sistema la ha procesado y mostrado sus resultados.

Podríamos imaginar incluso una siruación peor, en la que las dos últimas respuestas llegaran en orden inverso: Primero las sugerencias de ab y luego las de abu. Ésto llevaría a la aplicación a un estado WRONG persistente hasta la siguiente interacción del usuario.

Código

Este es el código implicado en el tratamiento del evento de actualización y del evento de recepción de la repuesta:

/src/container/SearchBoxContainerUnsafe.jsx
    handleCallback(result){ (1)
        const suggestions = result.map(suggestion_json => suggestion_json.value);
        this.setState({suggestions: suggestions});
    }

    handleChange(event) { (2)
        if (event.target.value) { (3)
            suggest(event.target.value) (4)
                .then(response => response.json()) (5)
                .then(this.handleCallback);

            this.setState({value: event.target.value}); (6)
        } else {
            this.setState({value: "", suggestions: []});
        }

    }
1 Es la función que se ejecutará cuando se termine de parsear la respuesta.
2 Es la función que se ejecuta al cambiar el valor del text-box. Inicia el fetch.
3 Comprueba que exista valor de text-box. Si es vacía, vacía las sugerencias y no lanza fetch.
4 El método suggest() es una abstracción de la API. Es una buena práctica en Javascript externalizar las llamadas a API en funciones que devuelvan el Promise que devuelve fetch.
5 La función .then() define qué función callback se ejecutará cuando acabe termine la Promise. En este caso, cuando termine el fetching se ejecutará response.json() que devuelve otro Promise. Cuando éste último se complete, se ejecutará nuestro handleCallback().
6 Se fija el estado para value para que el text-box muestre el valor insertado.
Como se ha mencionado, .then() requiere una función como parámetro. En ambos casos vemos dos formas de proveerle una. En el primero, usamos una función anónima, también llamadas lambda (λ), definida usando la sintaxis de ECMAScript6 (parámetro) ⇒ elementro-de-respuesta. En el segundo caso pasamos una función con nombre, creada por nosotros. Es importante entender que para pasar una función, escribimos su nombre sin paréntesis () ya que si ponemos sus paréntesis, la estaremos ejecutando/evaluando y lo que pasaremos a .then() será el resultado de esa evaluación/ejecución.

Searchbox Safe 1

Para solventar el problema detectado en la solución anterior, hay que adjuntar a la función de respuesta (callback) información sobre el estado en el fue generada esa petición. Posteriormente, en el momento en el que se recibe la respuesta, esa función callback debe discernir si la respuesta aún es válida en base a:

  • El estado actual

  • La información sobre el estado anterior

  • La respuesta

Para pasar esa información, crearemos una función que devuelva otra función:

/src/container/SearchBoxContainerSafe1.jsx
    handleCallback(result){ (1)
        const suggestions = result.map(suggestion_json => suggestion_json.value);
        this.setState({suggestions: suggestions});
    }

    checkValue(eventValue){ (2)
        return eventValue === this.state.value;
    }

    thenBuilder(value) { (3)
        return response => { (4)
            if (this.checkValue(value)) {
                response.json().then(this.handleCallback)
            }
        }
    }

    handleChange(event) {
        if (event.target.value) {
            suggest(event.target.value)
                .then(this.thenBuilder(event.target.value)); (5)

            this.setState({value: event.target.value});
        } else {
            this.setState({value: "", suggestions: []});
        }

    }
1 La función es idéntica la caso anterior.
2 Comprueba que el valor que recibe y el valor del estado son iguales. Así se asegura consistencia.
3 Esta es la nueva función que resuelve el problema. Toma como parámetro un valor, que en este caso es el valor por el cual se van a pedir las sugerencias. Devuelve una función de un parámetro. (Igual que la función que requiere .then())
4 Esa función devuelta, tiene como parámetro el response de la petición y su lógica consiste en que, si el chackValue() es correcto, se continúa el proceso y si no, se aborta.
5 Dentro del primer .then() se ejecuta/evalua thenBuilder() pasándole el valor del text-box. La función que se pasa a .then() es la devuelve esta ejecución/evaluación, no thenBuilder() en sí.

De éste modo, un tanto farragoso, se ha implementado una solución robusta que solventa los errores al repetir los pasos de la solución insegura.

Si no se ha comprendido bien el funcionamiento de esta solución, tal vez ayude observar la siguiente y después volver a ésta.
Para endurecer aún más la seguridad, sería deseable establecer un timeout y controlar la lógica en caso de error (4XX o 5XX).

Searchbox Safe 2

El principal error de la solución anterior es su verbosa forma de aplicar una técnica muy conocida en la programación funcional: El currying.

El currying básicamente implica definir una función que devuelve otra función. La primera función toma un(os) parámetro(s) y los usa para modelar segunda función, la devuelta.

El lenguaje usado, ECMAScript6, soporta nativamente una sintaxis específica para currying de modo que su aplicación es tremendamente ligera en términos de verbosidad.

En ES6 podemos definir:

// Función anónima
() => valor-de-return;

// Función con nombre
nombre_de_la_funcion(){
  return valor_de_return
};
// Forma anónima equivalente (a la que se le da nombre después) (ES6 style)
nombre_de_la_funcion = () => valor_de_return;

// Función con nombre que devuelve función anónima (currying)
nombre_de_la_funcion(arg1){
  return (arg2) => <código que usa arg1 y arg2>
};

// Forma anónima equivalente (a la que se le da nombre después)
// que devuelve función anónima  (ES6 style)
nombre_de_la_funcion = (arg1) => (arg2) =>  <código que usa arg1 y arg2>;

Cualquiera de las dos sintaxis que usemos en el currying nos dará el mismo resultado, y podrá ser llamado:

nombre_de_la_funcion(arg1);
//return: función2

nombre_de_la_funcion(arg1)(arg2);
//return: El resultado de <código que usa arg1 y arg2>

Sabiendo ésto, hemos aplicado ésta técnica para simplificar el código combinando thenBuilder() y handleChange() en una sola función:

/src/container/SearchBoxContainerSafe2.jsx
    checkValue(eventValue){
        return eventValue === this.state.value;
    }

    handleCallback = (value) => (response) => { (1)
        if (this.checkValue(value)) {
            response.json().then( (result) => (2)
                this.setState(
                    {suggestions: result.map(suggestion_json => suggestion_json.value)}
                )
            )
        }
    };

    handleChange(event) {
        if (event.target.value) {
            suggest(event.target.value)
                .then(this.handleCallback(event.target.value));

            this.setState({value: event.target.value});
        } else {
            this.setState({value: "", suggestions: []});
        }
    }
1 Definición de la nueva función, utilizando la sintaxis de ES6.
2 Aquí es donde se anidan lo que antes eran dos funciones. Esta función anónima es la que antes se llamaba handleChange(). De este modo queda más compacto ya que el propio result.map() se hace dentro del .setState() sin usar variables intermedias.
