En este documento se estudiarán diversos aspectos sobre persistencia en Docker.

Pre-requisitos

  • Docker 17.12.0-ce

  • Docker Compose 1.19.0

Introducción

Como es sabido, Docker basa su comportamiento en contenedores, pero no es tan habitual entender el funcionamiento interno de éstos.

Imágenes y Contenedores

De forma análoga a como funciona cualquier máquina (física o virtual), todo parte de una imagen. Una imagen no es otra cosa que un sistema de ficheros que refleja el estado de un sistema en un momento concreto.

En el caso de Docker, existen infinidad de imágenes disponibles para utilizar desde su repositorio oficial. Entre ellas se encuentran desde la imagen mínima, scratch hasta imágenes más complejas con sistemas ya instalados como debian, ubuntu o alpine o incluso con software específico o librerías y dependencias ya preconfiguradas como node8. Podríamos decir que una imagen es básicamente un sistema de ficheros con sus directorios y documentos empaquetados de una determinada manera.

Todas estas imágenes se pueden encontrar en el Repositorio oficial de Docker Hub.

Capas

La diferencia entre una imagen y otra viene dada por las capas (layers) que la conforman. Cada capa es un diff del sistema de ficheros de la imagen anterior. Si partimos de una imagen base y en una capa añadimos un fichero helloword.txt, la capa que hemos creado tan sólo ocupara en memoria el tamaño de ese fichero.

Existen varias formas de crear una nueva capa (a.k.a. hacer cambios en el sistema de ficheros):

  • Haciendo un cambio controlado y programado en un script Dockerfile.

  • Accediendo a través de la terminal al contenedor, realizando los cambios por línea de comandos y guardando los cambios en una nueva imagen con docker commit.

Por cuaquiera de las dos vías, el resultado será el mismo: una nueva imagen que compartirá todas las capas de su imagen base. Ésto es interesante porque Docker lo aprovechará para minimizar el espacio que ocupan varias imágenes.

Cabe destacar que en un Dockerfile, cada instrucción conforma una nueva capa y no el resultado completo de la ejecución por lo que en el ejemplo mencionado suponemos que sólo se ejecuta una instrucción como ADD . /opt/app

Instancias

Si una imagen es un sistema de ficheros que refleja el estado de un sistema, un contenedor es una instancia "en ejecución" de esa imagen.

El entrecomillado se debe a que un contenedor puede ser detenido y vuelto a poner en parcha, pero a efectos prácticos podemos diferenciar contenedores e imágenes en éstos términos.

Parece entonces que una imagen tan sólo es una plantilla para instanciar un contenedor, hasta que observamos la implementación interna en la documentación sobre persistencia.

En ella se puede observar cómo un contenedor no es sino una capa extra sobre una imagen ya existente, con una serie de metadatos y configuraciones adjuntos. Por esto mismo es tan sencillo realizar docker commit. Porque cuando ejecutamos dicho comando, tan sólo estamos estamos convirtiendo esa capa de lectura/escritura a sólo lectura.

Las capas R/W están almacenadas en /var/lib/docker/containers mientras que las capas RO están en /var/lib/docker/{nombre_del_driver}. Ésto se puede comprobar en esta demo de su documentación oficial.

Cada cambio que hacemos en un contenedor en ejecución es en realidad hecho sobre el sistema de ficheros en el que están alojada su capa R/W utilizando además una política copy-on-write.

Cada driver ejecuta esta política de forma diferente. Para más información, consultar la documentación.

Este proceso tiene un impacto en el rendimiento del sistema, ese fue uno de los motivos para crear los volúmenes, para casos en los que se necesita un IO intensivo y eficiente y no se necesita que formen parte del sistema de capas.

Volúmenes

Como cita la documentación sobre el comando docker commit:

The commit operation will not include any data contained in volumes mounted inside the container.

— Documentación Oficial de Docker

Los volúmenes son introducidos en docker para emular el funcionamiento de la herramienta mount de unix. De ese modo, se crea una unidad de almacenamiento (el volumen) y se monta en la ruta especificada.

Esta unidad de almacenamiento no estará sujeta al sistema de layers del contenedor, por lo que las operaciones de I/O serán mucho más eficientes.

No hay que confundir volúmenes con bindings. Aunque su sintaxis sea parecida (bind_host:bind_container y volume:mount_container) y su comando el mismo (-v), un binding monta un directorio local en un directorio del contenedor mientras que el volumen es un elemento creado y controlado por Docker cuya hubicación en el host no es necesario conocer.

Docker Volume Manager

Es un contenedor Ultra Light Weight diseñado para exportar e importar volúmenes docker en paquetes así como explorar volúmenes existentes.

Funcionalidades

Las funcionalidades que ofrece éste comando son:

  1. Exportar un volumen existente a un fichero comprimido

  2. Importar volumen comprimido a un volumen existente

  3. Copiar un volumen a otro con otro nombre

  4. Explorar los ficheros dentro de un volumen

  5. Escribir por pantalla el contenido de un fichero dentro de un volumen

La imagen exportada coincide con el nombre del volumen fuente al exportar y debe coincidir con el nombre del volumen destino al importar. La explicación es que si se quiere duplicar un volumen, existe el comando copy mientras que export/import están pensados para backup/restore o para migrar de máquina.

Comandos

Comandos nativos

Primero, es necesario conocer los comandos nativos que sí soporta Docker para la gestión de imágenes.

Listado de volúmenes
docker volume ls
Creación de un volumen
docker volume create VOLUME
Borrado de un volumen
docker volume rm VOLUME

Comandos de Docker Volume Manager

El resto de funcionalidades no están soportadas y será necesario para ello usar la herramienta.

Exportar un volumen existente a un fichero comprimido
./docker-volume export SOURCE_VOLUME

Creará un fichero comprimido en el directorio desde donde se ejecute el comando (pwd) llamado SOURCE_VOLUME.tar que contendrá el volumen SOURCE_VOLUME comprimido.

Éste comando funciona aunque el volumen esté en uso.
Importar volumen comprimido a un volumen existente
./docker-volume import DESTINATION_VOLUME

Tratará de leer el fichero comprimido llamado llamado SOURCE_VOLUME.tar que estará en el directorio desde donde se ejecute dicho comando (pwd) e importará su contenido en el volumen SOURCE_VOLUME.

Este comando funciona aunque el volumen esté en uso. El/los contenedor/es que estén usando ese volumen, percibirán el import como un simple cambio de los ficheros en él.
Copiar un volumen a otro con otro nombre
./docker-volume copy SOURCE_VOLUME DESTINATION_VOLUME

Copiará el contenido de el volumen SOURCE_VOLUME al volumen DESTINATION_VOLUME.

Si no existe el volumen, este comando lo crea. El/los contenedor/es que estén usando el volumen de destino volumen, percibirán el copy como un simple cambio de los ficheros en él.
Explorar los ficheros dentro de un volumen
./docker-volume ls VOLUME PATH

Muestra los ficheros y directorios en la ruta PATH dentro del volumen VOLUME.

Si bien es cierto que puede ser más cómodo hacer un export y abrir el fichero coprimido, ésta funcionalidad permite ejecutar un comando ls sobre cualquier ruta dentro del volumen.
Escribir por pantalla el contenido de un fichero dentro de un volumen
./docker-volume cat VOLUME PATH

Envía a la salida estándar el resultado de la ejecución del comando cat sobre el fichero del volumen PATH.

Esa salida puede usarse en una pipe en el host encadenada con comandos como grep, > out.txt, tr, sed, etc.

Funcionamiento interno

La clave del éxito de esta herramienta es el uso de la imagen ultra ligera busybox, que contiene un set de las herramientas más comunes de unix simplificadas que no requieren de un sistema operativo.

Gracias a esa imagen, podemos crear un contenedor ad-hoc al cual le montamos el volumen con el que deseamos operar para luego ejecutar un comando de bussybox sobre él.

Internamente, todos los comandos internos usan la estructura CABECERA_COMÚN [VOLúMENES_EXTRA] busybox COMANDO.

Cabecera común a todos los comandos internos
docker run \ (1)
--rm \ (2)
-v $2:/vol:ro (3)
1 Comando que descarga (si no existe ya) la imagen dada y crea y ejecuta un contenedor de ella.
2 Argumento que estipula que el contenedor será borrado al finalizar su ejecución. Imprescindible para evitar la acumulación de contenedores usados.
3 Montaje del volumen llamado igual al segundo argumento ($2) al punto /vol en el contenedor.
La anotación :ro del contenedor define que será de sólo lectura, lo que garantiza que la imagen original no será modificada y mucho menos dañada. IMPORTANTE: El comando import sí requiere permisos de escritura sobre el volumen original por motivos evidentes, por lo que prescinde de esta opción igual que el comando copy para su volumen destino.
export
        docker run --rm -v $2:/vol:ro \
          -v $(pwd):/ext \
          busybox \
          tar cvf /ext/$2.tar /vol/;  (1)
1 Hace el binding el directorio local del host y el directorio /ext del contenedor.
2 Ejecuta el comando de comprimir /vol en un un .tar y guardarlo en el directorio bindeado /ext.
import
        docker run --rm -v $2:/vol \
          -v $(pwd):/ext \
          busybox \
          tar xvf /ext/$2.tar --strip 1 -C /vol;  (1)
1 Hace el binding el directorio local del host y el directorio /ext del contenedor.
2 Ejecuta el comando de descomprimir el contenido del .tar en el directorio del volumen /vol.
copy
        docker run --rm -v $2:/vol:ro \
          -v $3:/newvol \
          busybox \
          cp -a /vol/. /newvol/;  (1)
1 Monta el segundo volumen en el directorio /newvol. contenedor
2 Ejecuta el comando de copiar todo el contenido del directorio del volumen /vol al directorio del nuevo volúmen /newvol.
ls
        docker run --rm -v $2:/vol:ro \
          busybox \
          ls /vol/$3;  (1)
1 Ejecuta el comando ls dentro del directorio /vol con una ruta definida en $3.
cat
        docker run --rm -v $2:/vol:ro \
          busybox \
          cat /vol/$3;  (1)
1 Ejecuta el comando cat dentro del directorio /vol con una ruta definida en $3.

Usos creativos de las funcionalidades

A partir de las funcionalidades básicas se pueden extraer otros usos creativos de esta herramienta:

Utilizar volúmenes plantilla

Aprovechando la capacidad de copiar volúmenes se pueden crear una serie de volúmenes plantilla y hacer copias de ellos.

Por ejemplo:

  1. Tenemos una estructura de carpetas en el host local.

  2. Creamos un volumen nuevo con docker volume create <nombre_del_volumen>. (Se puede seguir algún principio de nombrado siempre que docker lo permita. Ej: t_<nombre_de_la_platilla>)

  3. Comprimimos la estructura en <nombre_del_volumen>.tar. (El .tar ha de contener todo en una carpeta raíz).

  4. Importamos el contenido del .tar al volumen con ./docker-volume import <nombre_del_volumen>.

Migrar de máquina contenedores con sus datos

Al exportar un contenedor con docker save los volúmenes no son inluídos. Este herramienta permite hacer migraciones completas sin perder el estado de los mismos.

Por ejemplo:

  1. Paramos el contenedor con docker stop <nombre_del_contenedor>.

  2. Listamos sus volúmenes con docker inspect <nombre_del_contenedor> (sección "Mounts").

  3. Guardamos el contenedor como imagen con docker commit <nombre_del_contenedor> <nueva_tag>.

  4. Guardamos la nueva imagen con docker save <nueva_tag> -o <nombre_del_contenedor>.tar.

  5. Exportando sus volúmenes ./docker-volume export <nombre_del_volumen>.

  6. Movemos los ficheros resultantes ( contenedor.tar, vol1.tar, vol2.tar, …​ ) a la máquina destino.

  7. Creamos los volúmenes en la máquina destino con docker volume create <nombre_del_volumen>.

  8. Importamos los volúmenes en la máquina destino con ./docker-volume import <nombre_del_volumen>.

  9. Cargamos la imágen en la máquina destino con docker load -i <nombre_del_contenedor>.tar.

  10. Creamos un nuevo contenedor vinculando esos volúmenes a sus puntos originales con docker run <nueva_tag> -v <nombre_del_volumen>:<punto_de_montaje>.

Los pasos 3. y 4. podrían resumirse en uno sólo: docker export <nombre_del_contenedor>. El fichero resultante sería más pequeño, pero habríamos perdido las capas y los metadatos de la imagen. A pesar de que resulta tentados ahorrarnos ese espacio, en la máquina destino, puede que muchas de las capas usadas por la imagen ya existieran por lo que al importarse en ella, el espacio que ocupará en disco será menor que con export. En otras palabras, comit && save guarda una imagen multicapa y export guarda una imagen monocapa.
